## 1. 我这个项目到底是什么

### 1.1 一句话理解
这是一个面向学生的 **AI IELTS Speaking 练习与评分产品**，同时带有一个轻量后台，用来做结果复核、异常处理和规则治理。

### 1.2 不要把它理解成什么
我不能把它只理解成：
- 一个 AI 聊天机器人
- 一个普通语音对话工具
- 一个只靠大模型“自由发挥”的产品

### 1.3 它真正的本质
这个项目本质上是在做一个闭环：

**题目 → 学生作答 → 语音转文字 → AI 评分 → 分维度反馈 → 历史记录 → 异常复核 → 规则优化**

所以我做的不是单点功能，而是一套能真正跑起来的教育场景 AI 流程。

### 1.4 这个项目最重要的价值
核心价值不是“用了 AI”，而是：
- 让学生更容易开口练
- 让反馈更结构化、更有帮助
- 让结果更可控、更稳定
- 当结果有争议时，平台可以复核和处理

### 1.5 当前阶段必须牢记的定位
当前阶段不是做“大而全平台”，而是先做一个 **真实可用的 Web MVP**。

MVP 的目标不是完美，而是验证：
- 学生是否愿意使用
- AI 反馈是否有帮助
- 结果是否足够可信
- 平台是否具备最小治理能力

## 2. 我最需要建立的思路

这一章最重要。技术以后都可以慢慢补，但如果思路错了，项目会一直乱。

### 2.1 MVP 思路：先做最小可用，不一步到位
我要时刻问自己：
- 这个功能是不是第一版必须有？
- 如果没有它，用户还能不能完成一次核心练习？
- 如果能，那它大概率不是当前最高优先级

当前第一版最重要的不是“功能多”，而是“闭环通”。

#### 我要记住的关键词
- 最小可用
- 验证价值
- 跑通闭环
- 延后非核心需求

- 什么是 must-have / should-have / nice-to-have

所以这个项目要优先做的是：

- 反馈可解释

## 4. 我必须掌握的 AI 知识点

### 4.1 Prompt 工程
这是我当前最核心的 AI 能力之一。

我不能只会写“请给这个回答打分”。
我要学会把 prompt 写成一个可控、稳定、能落地的规则描述。

#### 我需要会的内容
- 给模型明确角色
- 给模型明确任务目标
- 给模型明确评分维度
- 给模型明确输出格式
- 给模型明确边界条件
- 给模型明确异常标记逻辑

#### 在项目里的具体表现
不是只写：
- “请帮我评价一下这段回答”

而是写成：
- 按 IELTS speaking 维度打分
- 返回总分和子分
- 给出问题定位
- 给出改进建议
- 用固定 JSON 返回
- 当内容不足或识别异常时标记风险

---

### 4.2 结构化输出
AI 要真正接入产品，必须尽量结构化。

因为只有结构化，结果才能：
- 保存到数据库
- 稳定展示在前端
- 被后台筛选和复核
- 做历史对比和统计

#### 我要掌握
- JSON 输出
- 字段设计
- schema 校验
- 为什么不能只返回一大段自然语言

#### 项目里可能的结果结构
- totalScore
- fluencyCoherence
- lexicalResource
- grammaticalRangeAccuracy
- pronunciation
- completeness
- summaryFeedback
- detailedFeedback
- riskFlag
- riskReason

---

### 4.3 评分维度理解
我要逐步理解 IELTS speaking 评分维度，不然就很难设计好 AI 评分和反馈。

#### 当前核心维度
- 流利度与连贯性
- 词汇
- 语法
- 发音
- 完整度

#### 我要掌握
- 每个维度在看什么
- 每个维度容易出现什么问题
- AI 容易误判什么
- 如何把抽象维度翻译成用户看得懂的反馈

---

### 4.4 模型边界意识
我要明白：
- 模型不是绝对准确
- 模型会受 prompt、输入文本质量、ASR 结果影响
- 模型可能前后不一致
- 模型可能给出看起来合理但并不可靠的结论

#### 我要掌握
- 什么叫 hallucination
- 什么叫模型不稳定
- 什么叫上下文依赖
- 为什么输入质量会影响输出质量

---

### 4.5 AI 评估意识
好的 AI 功能不是凭感觉判断的。

我要逐步学会：
- 准备测试样本
- 比较不同 prompt 版本结果
- 看哪些 bad case 经常出现
- 记录规则版本
- 做简单评测

#### 我要掌握
- 什么是 prompt 迭代
- 什么是样本集测试
- 什么是 bad case 回放
- 为什么要做版本管理

## 8. 我需要记住的关键判断标准

以后做方案、看需求、学技术时，我都可以拿这几条来判断是否值得当前投入。

### 8.1 它是否直接服务于 MVP 主闭环
如果一个东西不能帮助用户更顺畅地完成：
- 练习
- 评分
- 反馈
- 历史查看

那它短期优先级通常不高。

### 8.2 它是否提升结果的可信度
如果一个东西能提升：
- 稳定性
- 可解释性
- 一致性
- 可复核性

它通常值得优先考虑。

### 8.3 它是否只是“看起来高级”
很多技术点听起来很厉害，但不一定适合当前阶段。

我要时刻提醒自己：
- 现在需要的是可用 MVP，不是技术炫技
- 先把核心流程做对，再考虑复杂优化

### 8.4 它是否能帮助我真正理解项目
因为这是一个“边做边学”的项目，所以我不仅要做出来，还要能说清楚：
- 为什么这么设计
- 为什么优先做这个
- 这个模块在整体系统里起什么作用

---

## 9. 我当前阶段建议掌握的技术栈理解

这里不是要求我全部精通，而是要知道为什么选它们。

### 9.1 Web 端推荐理解路线
- React：做页面组件和交互
- Next.js：把前端页面和后端接口放在一个工程里，更适合 MVP
- Tailwind CSS：提高页面搭建速度
- Supabase / PostgreSQL：存用户、练习、评分、复核等数据
- ASR 服务：把语音转成文字
- LLM API：做评分与反馈生成

### 9.2 为什么这条路线适合当前阶段
因为它：
- 学习路径相对清晰
- 社区资料多
- 适合快速做 Web MVP
- 能比较顺畅地把前端、后端、数据库、AI 串起来

## 10. 我应该怎样学习，才不会乱

### 10.1 用“项目驱动学习”，不要脱离场景死学
我不需要先学完整前端课程、完整后端课程、完整 AI 理论，再开始做项目。
更合适的方法是：
- 遇到问题再学对应知识
- 学完马上用到项目里
- 用项目反过来巩固知识

### 10.2 每次只解决一个层级的问题
比如：
- 当前先搞清楚页面流程，就不要同时钻数据库优化
- 当前先搞清楚 prompt 输出，就不要同时研究太深的模型原理

### 10.3 始终问“这个知识点在项目里对应什么”
如果一个知识点不能对应到项目里的模块、流程或问题，我暂时不用学太深。

### 10.4 学完要能复述
真正掌握一个知识点，不是“看过”，而是能用自己的话说清楚：
- 它是什么
- 为什么需要它
- 它在项目里的位置是什么
- 如果没有它会怎样

## 13. 按产品 / 前端 / 后端 / AI 拆分的阶段学习路线

这一章是给初学者用的。
目标不是让你一次把所有东西学完，而是让你知道：
- 现阶段先学什么
- 每一类内容学到什么程度就够推进项目
- 每一块知识在项目里具体对应什么

### 13.1 总原则：四条线并行，但主次分明
这个项目建议同时保留四条学习线：
- 产品线：保证方向不跑偏
- 前端线：保证页面和交互能做出来
- 后端线：保证业务链路能跑通
- AI 线：保证评分和反馈能工作且尽量稳定

但不是四条线平均发力，而是按阶段切换重点。

建议顺序是：
1. 先打通产品理解
2. 再建立前端基础感知
3. 再理解后端和数据怎么串联
4. 再重点打磨 AI 评分链路
5. 最后补治理、评估、优化能力

---

### 13.2 第一阶段：先建立产品脑图
这个阶段最重要，不要急着写很多代码。

#### 产品线要学什么
- 用户是谁
- 用户为什么来
- 用户完成一次练习的完整流程是什么
- MVP 为什么只做这些功能
- 学生端和后台端分别解决什么问题

#### 前端线要学什么
- 页面是什么
- 一个产品最少需要哪些页面
- 页面之间如何跳转
- 一个页面上有哪些基础交互元素

#### 后端线要学什么
- 前端为什么需要后端
- 一个练习请求是如何被接收和处理的
- 什么叫接口
- 什么叫请求和响应

#### AI 线要学什么
- AI 在项目里负责什么，不负责什么
- 为什么 AI 不是整个产品
- 为什么需要 prompt
- 为什么需要结构化输出

#### 这一阶段学完后的目标
你至少要能用自己的话说清楚：
- 这个产品在解决什么问题
- MVP 为什么这样定
- 用户完成一次练习会经过哪些步骤
- AI 在整个链路中的位置是什么

#### 这一阶段不需要过度深入的东西
- 不用急着学复杂数据库设计
- 不用急着研究高深模型原理
- 不用急着做复杂后台权限系统

---

### 13.3 第二阶段：前端入门，先把页面和交互做出来
当你已经知道产品流程后，下一步就是让用户“看得见、点得动”。

#### 这一阶段前端是主线

#### 产品线要配合理解什么
- 哪些页面是 MVP 必须有的
- 每个页面的目标是什么
- 页面里哪些信息最重要
- 用户在哪些地方容易卡住

#### 前端线重点学什么
- HTML / CSS / JavaScript 基础概念
- React 的组件思想
- state 是什么
- props 是什么
- 表单输入和按钮点击怎么处理
- 列表渲染
- 条件渲染
- 页面加载中、成功、失败三种状态怎么表现

#### 在项目里对应什么
你可以先理解或实现这些页面：
- 练习开始页
- 题目展示页
- 录音区域
- 结果展示页
- 历史记录页
- 后台会话列表页

#### 后端线在这一阶段只要理解到什么程度
- 知道前端最终要调用 API
- 知道页面上的数据有些来自本地状态，有些来自服务端
- 暂时不要求自己把后端细节完全吃透

#### AI 线在这一阶段只要理解到什么程度
- 知道评分结果未来会以 JSON 或结构化数据展示在页面上
- 知道页面展示依赖 AI 输出字段是否稳定

#### 这一阶段学完后的目标
你至少要能理解：
- 一个 React 页面是怎么组织的
- 一个按钮点击后，页面状态为什么会变化
- 结果页为什么需要结构化数据来渲染

---

### 13.4 第三阶段：后端入门，先把数据链路串起来
当前端页面有了基本感觉后，就要开始理解“数据是怎么流动的”。

#### 这一阶段后端是主线

#### 产品线要配合理解什么
- 一次练习提交后，系统内部到底发生了什么
- 哪些数据必须保存
- 为什么历史记录和复核能力依赖数据库

#### 前端线在这一阶段要理解什么
- 如何发请求
- 如何把表单、文本、音频文件提交给后端
- 如何接收后端返回结果并展示

#### 后端线重点学什么
- HTTP 基础
- GET / POST 的区别
- API Route 是什么
- 请求体和响应体是什么
- 文件上传是怎么回事
- 后端如何调用外部服务
- 后端如何做错误处理

#### 数据库线一起补什么
- 表、字段、记录、主键这些基础概念
- 一次练习至少要保存哪些数据
- 历史记录为什么需要 session 设计
- 评分结果为什么要单独存

#### 在项目里对应什么
你要逐步能理解或实现：
- 提交录音接口
- 转写接口或转写流程
- 评分接口
- 保存练习记录
- 查询历史记录
- 查询后台会话详情

#### AI 线在这一阶段要理解什么
- 大模型调用其实也是后端在请求一个外部服务
- AI 结果返回后，需要经过后端整理再交给前端

#### 这一阶段学完后的目标
你至少要能说清楚：
- 用户点一次“提交”后，前后端各做了什么
- 为什么数据库是必须的
- 为什么历史记录、复核、版本管理都离不开后端

---

### 13.5 第四阶段：AI 评分链路专项学习
到这个阶段，才开始把重心放到 AI 本身。

#### 这一阶段 AI 是主线

#### 产品线要配合理解什么
- 用户最关心的不是模型多聪明，而是结果有没有帮助
- 什么样的反馈才算真正可用
- 什么样的评分结果容易引发争议

#### 前端线要配合理解什么
- 评分结果页如何展示总分、子分和反馈
- 风险提示或异常提示怎么展示给用户或后台

#### 后端线要配合理解什么
- 如何调用模型 API
- 如何传 prompt
- 如何拿到结构化结果
- 如何处理格式错误、超时和失败

#### AI 线重点学什么
- Prompt 的基本结构
- 角色、任务、约束、输出格式
- 如何设计评分维度
- 如何要求 JSON 返回
- 如何做 riskFlag / riskReason
- 什么是 bad case
- 如何做 prompt 版本对比

#### 在项目里对应什么
你要逐步能理解或实现：
- 一版基础评分 prompt
- 一版结构化输出 schema
- 一版风险标记逻辑
- 一组测试样本验证 prompt 是否稳定

#### 这一阶段学完后的目标
你至少要能说清楚：
- 为什么 prompt 不能随便写
- 为什么结构化输出对产品很关键
- 为什么 AI 评分必须考虑异常和争议情况

---

### 13.6 第五阶段：治理与运营能力补齐
当前面基本能跑起来后，就进入你这个项目真正有差异化的部分。

#### 这一阶段产品与治理是主线

#### 产品线重点学什么
- 什么叫 AI 治理
- 为什么教育产品需要申诉、复核和质检
- 什么叫“可追责、可回看、可优化”

#### 前端线重点学什么
- 后台列表页和详情页如何设计
- 如何展示异常状态、复核状态、版本信息
- 如何给运营更高效的查看体验

#### 后端线重点学什么
- 如何设计 review_tasks / appeals / prompt_versions 等数据结构
- 如何支持筛选、查询、状态流转
- 如何记录人工复核结论

#### AI 线重点学什么
- 如何从 bad case 反推 prompt 优化
- 如何建立简单评测集
- 如何做版本迭代记录

#### 在项目里对应什么
你要逐步能理解或实现：
- 异常标记
- 人工复核记录
- 用户申诉入口
- prompt 版本追踪
- 后台筛选与回放

#### 这一阶段学完后的目标
你至少要能说清楚：
- 你的项目为什么不只是一个普通 AI 陪练工具
- 后台治理为什么是核心壁垒之一
- bad case 沉淀为什么重要

---

### 13.9 你现在最适合的学习策略
