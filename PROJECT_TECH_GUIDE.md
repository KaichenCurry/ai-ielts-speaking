# AI IELTS Speaking 项目技术说明（小白友好版）

## 0. 这份文档是干什么的
这是一份专门给 **技术初学者** 看的项目技术说明。

它主要回答这些问题：
- 这个项目现在到底用了哪些技术？
- 这些技术分别是干什么的？
- 为什么要这样设计，而不是随便拼起来？
- 学生端、后台端、AI、数据库之间是怎么串起来的？
- 像 **Supabase、ASR、Prompt、API、JSON Schema、bad case** 这些词到底是什么意思？

你可以把这份文档理解成：

**“这个项目的技术地图 + 项目运行逻辑说明 + 专业名词翻译本”**

---

## 1. 先用一句人话讲清楚这个项目
这个项目不是单纯的“AI聊天网站”。

它本质上是在做一条完整链路：

**学生说话 → 系统录音 → AI转文字 → AI评分 → 生成反馈 → 存入数据库 → 学生查看结果 → 后台复核异常 → 继续优化规则**

所以它不是一个单点功能，而是一套小型 AI 产品系统。

---

## 2. 这个项目当前的核心结构
目前项目可以分成 4 个大块来看：

### 2.1 学生端
学生用来练口语。

当前主要页面有：
- 首页
- 练习入口页
- 练习进行页
- 结果页
- 历史记录页
- 历史详情页

学生端的核心目标是：
**让用户完成一次完整练习，并拿到结构化反馈。**

---

### 2.2 后台治理端
后台不是给学生用的，是给平台/运营/老师/复核人员看的。

当前主要页面有：
- 后台总览页
- 会话列表页
- 会话详情 / 复核页
- Prompt / 规则版本页

后台的核心目标是：
**当 AI 结果有争议时，系统可以查看、标记、申诉、复核、沉淀问题样本。**

这也是这个项目和很多普通 AI 陪练产品不一样的地方。

---

### 2.3 AI 能力层
这里负责两件事：
1. 把学生的录音转成文字
2. 根据文字进行评分和反馈

当前项目里用了 OpenAI 的两个方向：
- 语音转写（ASR）
- 文本评分（LLM scoring）

简单理解：
- **ASR** = 听懂你说了什么
- **评分模型** = 读懂文字后给出评分和建议

---

### 2.4 数据层
这里负责把结果“存起来”和“取出来”。

当前项目的数据层主要分两种：
- **mock data（假数据）**：先演示流程，方便开发
- **Supabase（真实数据库）**：存真实练习记录、申诉状态、规则版本等

简单理解：
- mock data = 演示版临时假数据
- Supabase = 真正用于存储数据的后端服务

## 3. 这个项目当前用到了哪些技术

## 3.1 Next.js
### 它是什么
Next.js 是一个用来开发 Web 应用的框架。

### 你可以把它理解成
它像一个“网站开发总装厂”，帮你把：
- 页面路由
- 前端页面
- 后端接口
- 服务端渲染
- 项目结构

这些能力放到同一个工程里。

### 在这个项目里它负责什么
它负责整个项目的大框架：
- 页面怎么组织
- 路由怎么跳转
- API 接口怎么写
- 前端和后端代码怎么放在同一个项目里

### 为什么它适合这个项目
因为你这个项目不是单纯前端，也不是单纯后端，而是：
- 有页面
- 有接口
- 有数据库读写
- 有 AI 调用

Next.js 很适合这种“一个项目里把前后端都先搭起来”的 MVP。

---

## 3.2 React
### 它是什么
React 是一个前端 UI 库，用来做页面组件。

### 你可以把它理解成
React 就像搭积木。

比如：
- 一个按钮是一个组件
- 一个卡片是一个组件
- 一个评分面板是一个组件
- 一个录音工作区也是一个组件

最后把这些组件拼成完整页面。

### 在这个项目里它负责什么
它负责页面交互，比如：
- 按钮点击
- 表单填写
- 状态变化
- 结果展示
- 录音区域更新

---

## 3.3 TypeScript
### 它是什么
TypeScript 可以理解为“更严格的 JavaScript”。

### 它解决什么问题
普通 JavaScript 很自由，但容易写错。
TypeScript 会提前提醒你：
- 这里应该是字符串还是数字？
- 这个字段有没有漏？
- 这个对象结构对不对？

### 在这个项目里它为什么重要
因为你的项目里有很多结构化数据，比如：
- 分数结构
- 反馈结构
- 会话记录结构
- 申诉状态
- 复核状态
- Prompt 版本结构

TypeScript 可以帮你把这些数据“管整齐”。

### 一个直观例子
比如总分、流利度、词汇、语法、发音、完整度，这些字段必须统一。
否则前端显示和数据库字段很容易对不上。

---

## 3.4 Supabase（你说的“supabass”基本就是它）
### 它是什么
Supabase 是一个后端服务平台。

你可以把它理解成：

**“别人帮你搭好的一套云端后端基础设施”**

通常它可以提供：
- 数据库
- 身份认证
- 文件存储
- 后端接口能力

### 在你这个项目里，Supabase 主要是干什么的
当前项目里，它主要被当成：
- **数据库服务**
- **数据读写通道**

也就是说，练习结果、申诉状态、复核结果、规则版本、bad case 等，未来都可以放在 Supabase 里。

### 为什么不用自己从零搭数据库
因为 MVP 阶段最重要的是先把产品跑起来。
如果自己从零搭：
- 数据库
- 用户系统
- 权限系统
- 接口层

工作量会立刻变大。

所以 Supabase 很适合 MVP：
**省时间，够用，接入快。**

### 你可以把它记成一句话
**Supabase = 一个帮助你快速拥有“后端能力”的工具。**

---

## 3.5 OpenAI API
### 它是什么
API 可以理解成“程序和程序之间说话的接口”。

OpenAI API 的意思就是：
你的项目可以通过网络请求去调用 OpenAI 的模型能力。

### 在这个项目里用它做什么
主要做两件事：
1. 音频转文字
2. 根据文字评分

### 为什么不是直接在页面里写死 AI 结果
因为 AI 结果不是固定内容，而是每次根据用户输入动态生成的。
所以需要请求模型服务。

---

## 3.6 ASR
### 它是什么
ASR = Automatic Speech Recognition
中文一般叫：**自动语音识别**。

### 人话解释
就是：
**把人说的话，自动转成文字。**

### 在这个项目里为什么必须有它
因为学生说的是语音，但评分模型更容易基于文字来分析。
所以要先经过：

**语音 → 转写文字 → 再评分**

### 为什么现在先做“转文字再评分”
这是一个很重要的方法选择。
因为这样更容易：
- 看见输入内容
- 手动修正转写错误
- 保留文本记录
- 后台复核
- 做历史记录
- 做 bad case 沉淀

这就是所谓的：
**transcript-first（先文本后评分）**

---

## 3.7 Prompt
### 它是什么
Prompt 就是“你给 AI 的任务说明书”。

### 人话解释
不是随便问 AI 一句“你帮我评个分”就完了。
你需要明确告诉它：
- 你是谁
- 你要做什么
- 要按什么标准评
- 输出什么格式
- 哪些情况要标风险

### 在这个项目里 Prompt 为什么重要
因为教育场景最怕：
- 评分不稳定
- 反馈太空
- 前后不一致
- 模型自由发挥过多

所以 Prompt 的作用不是“让 AI 更会说话”，而是：
**约束 AI，让它更稳定、更可控。**

---

## 3.8 JSON Schema
### 它是什么
JSON 是一种结构化数据格式。

Schema 可以理解成“数据模板”或“数据规则”。

### 人话解释
你可以把 JSON Schema 理解成：

**“我提前规定 AI 返回结果必须长什么样。”**

比如必须返回：
- total
- fluencyCoherence
- lexicalResource
- grammar
- pronunciation
- completeness
- summary
- strengths
- priorities
- nextStep
- riskFlag
- riskReason
- confidence

### 它为什么重要
因为如果不限制，AI 很可能这次返回一个格式，下次又返回另一个格式。
那前端就很难稳定展示。

所以 JSON Schema 的作用是：
**让 AI 输出结构化、可读、可存、可展示。**

---

## 3.9 API Route / Route Handler
### 它是什么
这是 Next.js 里的后端接口写法。

### 人话解释
就是项目里的“后端入口”。
前端页面不直接去做所有事情，而是把任务交给这些接口。

### 在这个项目里有哪些典型接口
当前你这个项目里，已经能看到几类接口：
- 转写接口：接收音频，调用 OpenAI ASR
- 评分接口：接收文本，调用 OpenAI 评分
- 申诉接口：提交申诉备注
- 复核接口：更新治理状态
- 规则版本接口：新增版本或设为 current

### 为什么要这么做
因为：
- API key 不能暴露在前端
- 数据库操作应该放服务端
- AI 调用逻辑更适合放后端

---

## 3.10 MediaRecorder
### 它是什么
这是浏览器自带的录音能力。

### 人话解释
浏览器可以向用户申请麦克风权限，然后录音，最后得到一个音频文件。

### 在项目里它负责什么
学生点击“开始录音”后：
- 浏览器拿到麦克风权限
- 开始录音
- 停止后生成音频文件
- 上传给后端做转写

所以你可以把它理解成：
**学生语音进入系统的第一个入口。**

---

## 3.11 sessionStorage
### 它是什么
sessionStorage 是浏览器里的一种临时存储。

### 人话解释
它像一个“只在当前浏览器会话里暂存数据的小抽屉”。

### 在这个项目里为什么会用它
评分完成后，前端会先临时把结果放进去，这样结果页可以更快显示“刚刚生成的实时结果”。

同时，如果数据库里已经有记录，页面也可以回退展示数据库结果。

这是一种常见思路：
**先给用户快反馈，再靠数据库做稳定存档。**

---

## 3.12 mock data
### 它是什么
mock data 就是假数据、演示数据。

### 为什么要先用假数据
因为产品开发早期，很多真实能力还没完全接好，比如：
- 真实数据库
- 真实录音流程
- 真实评分结果
- 真实后台数据

如果什么都等真实接完再做页面，开发会很慢。

所以先用 mock data，可以先验证：
- 页面结构是否合理
- 用户流程是否顺
- 数据结构是否清楚

### 你要记住一句话
**mock 不是骗人，而是开发阶段常用的过渡方法。**

## 4. 这个项目当前采用了什么“方法”

## 4.1 MVP 方法
### 是什么
MVP = Minimum Viable Product
中文可以理解成：**最小可用产品**。

### 在这个项目里是什么意思
不是一上来做成超级完整平台，而是先保证：
- 学生能练一次
- 能拿到分数和反馈
- 能看到历史记录
- 后台能做最基本治理

### 为什么这很重要
因为现在第一目标不是“功能最全”，而是：
**验证这个产品闭环能不能跑起来。**

---

## 4.2 闭环方法
### 是什么
闭环的意思不是做一个页面，而是做完整流程。

### 这个项目的核心闭环
**开始练习 → 录音 → 转写 → 评分 → 结果展示 → 历史记录 → 异常治理**

### 为什么闭环比单点功能更重要
因为用户不会为“一个好看的页面”付费。
用户会为“真的完成了一次练习并得到有效反馈”付费。

---

## 4.3 transcript-first 方法
### 是什么
先把语音转成文本，再用文本评分。

### 为什么当前阶段这样做
因为它更适合 MVP：
- 更容易调试
- 更容易人工检查
- 更容易留历史记录
- 更容易做后台复核
- 更容易沉淀 bad case

### 缺点是什么
发音维度会有一定局限。
因为只看文本，无法完整反映真实发音。

### 这也是为什么代码里会提醒
“pronunciation 要谨慎判断，必要时降低 confidence（置信度）”。

---

## 4.4 结构化输出方法
### 是什么
不是让模型自由写一大段，而是要求它按固定结构返回。

### 为什么这样做
因为教育产品最怕结果乱。
结构化输出可以让：
- 前端容易展示
- 数据库容易存
- 后台容易复核
- 后续容易分析

---

## 4.5 mock-first，再逐步接真实能力
### 是什么
先用假数据把页面和流程跑起来，再逐步替换为真实能力。

### 这是好方法吗
是的，尤其适合你现在这种：
- 要做真实产品
- 但还在学习技术
- 又想保持推进速度

### 当前项目就明显采用了这个方法
你现在项目里很多地方是：
- 如果没配置 Supabase，就回退到 mock 数据
- 如果真实数据可用，就优先展示真实数据

这是一种非常典型的 MVP 开发方式。

---

## 4.6 治理优先的方法
### 是什么
不是只关心“AI 有没有生成结果”，还关心：
- 结果是否靠谱
- 有无争议
- 能不能申诉
- 能不能复核
- 能不能沉淀问题样本

### 为什么这在教育场景尤其重要
因为教育产品里，用户对错误反馈会特别敏感。
如果系统只会评分，不会治理，就很容易失去信任。

所以你这个项目的一个关键思路是：
**AI 输出不只是生成，还要可管理。**

---

## 5. 这个项目的业务逻辑怎么理解

## 5.1 学生端业务逻辑
学生端可以理解成 5 步：

### 第一步：选择练习类型
学生选择 Part 1 / Part 2 / Part 3。

### 第二步：开始录音
浏览器调用麦克风，录下学生回答。

### 第三步：音频转文字
系统把录音发给 ASR 服务，得到 transcript（转写文本）。

### 第四步：AI 评分
系统把题目、答题文本、时长等信息发给评分模型。
模型返回：
- 总分
- 分项分
- 总结反馈
- 优点
- 优先改进点
- 下一步建议
- 示例答案
- 风险标记
- 置信度

### 第五步：展示结果并保存
系统把结果展示给学生；如果数据库可用，就把数据存起来，供历史记录和后台治理使用。

---

## 5.2 后台治理逻辑
后台可以理解成 4 步：

### 第一步：查看会话
后台查看某条练习记录：
- 题目
- 转写
- 分数
- 反馈
- 风险状态

### 第二步：判断是否异常
如果发现有问题，可以：
- 标记风险
- 写风险原因
- 修改申诉状态
- 填写复核结果

### 第三步：沉淀 bad case
如果这条记录很典型，值得后续优化，就可以沉淀为 **bad case**。

### 第四步：关联规则版本
把 bad case 跟 Prompt / 规则版本关联起来，帮助后续分析：
- 是哪个版本容易出问题？
- 改完后是否变好？

---

## 6. 什么是 bad case
### 它是什么
bad case 直译是“坏案例”或“问题样本”。

### 人话解释
就是：
**系统这次表现得不够好，这条记录值得单独记下来，后面拿来复盘和优化。**

### 典型 bad case 例子
比如：
- 分数明显不合理
- 反馈太空泛
- 文本太短却给了高分
- 明显答非所问却没被识别
- 风险没标出来

### 为什么它很重要
因为 AI 产品优化，不能只靠感觉。
要靠问题样本持续积累。

所以 bad case 是一种非常重要的 AI 产品治理资产。

---

## 7. 什么是规则版本 / Prompt 版本
### 它是什么
就是记录“你当前到底用的是哪一套 AI 评分规则”。

### 为什么要版本化
因为 Prompt 是会变的。
今天你可能用 v0.1，明天可能改成 v0.2。

如果不做版本记录，你以后就很难回答：
- 为什么今天的结果和上周不一样？
- 哪一版更稳定？
- 哪一版导致了更多 bad case？

### 所以版本管理的作用是
**让 AI 规则变化可追踪。**

---

## 8. 风险标记、申诉、复核分别是什么意思

## 8.1 风险标记（risk flag）
意思是：
**这条结果可能不太可靠，需要注意。**

例如：
- 回答太短
- 内容太模糊
- 很难判断发音
- 模型信心不足

---

## 8.2 申诉（appeal）
学生觉得这次结果有问题，可以提交说明。

比如：
- 我觉得分数不合理
- 转写有误
- 反馈和我真实表现不符

---

## 8.3 复核（review）
后台人工或运营人员重新看这条结果，给出判断。

也就是：
**AI 先判一次，人再做二次判断。**

这是教育产品很重要的可信机制。

---

## 9. 为什么这个项目需要数据库，而不是只在前端展示
如果只是前端展示，当页面一刷新，很多数据就没了。

但你的项目需要：
- 历史记录
- 申诉状态
- 复核结果
- bad case 沉淀
- 规则版本追踪
- 后台统计

这些都说明：
**这个项目不是一次性网页，而是一个需要“长期记忆”的系统。**

数据库的意义就在这里。

## 10. 当前项目里有哪些“真实能力”，哪些还是“过渡方案”

## 10.1 已经接入或部分接入的真实能力
当前项目已经不是纯静态页面了，已经有一些真实能力：
- 浏览器录音
- OpenAI ASR 转写
- OpenAI 文本评分
- Supabase 数据读写（配置后可用）
- 学生申诉提交
- 后台复核更新
- Prompt 版本管理
- bad case 沉淀

---

## 10.2 仍然存在的过渡方案
当前仍保留一些 MVP 阶段很正常的过渡设计：
- 未配置 Supabase 时回退到 mock 数据
- 发音分仍主要基于文本推断，不是完整音频评分

**先跑通，再增强。**

## 13. 初学者最容易混淆的几个词

### 13.1 前端
用户看得见、点得到的页面部分。

### 13.2 后端
处理数据、调模型、写数据库的服务端部分。

### 13.3 接口 / API
前端和后端、你的系统和 OpenAI 之间“沟通的门”。

### 13.4 数据库
长期存数据的地方。

### 13.5 模型
负责理解或生成内容的 AI 能力核心。

### 13.6 Prompt
给模型下指令的说明书。

### 13.7 ASR
把语音转成文字。

### 13.8 Schema
规定返回数据长什么样。

### 13.9 Mock
开发阶段临时模拟的数据或流程。

### 13.10 Fallback
回退方案。
意思是：如果真实能力不可用，就先退回到备用方案。

例如：
- 没配置数据库 → 回退到 mock 数据
- 实时结果没拿到 → 回退展示持久化结果
