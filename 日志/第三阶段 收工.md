## 2. 已解决的问题
- 学生端申诉状态不透明。
  - 现象：学生提交申诉后，只能看到基础提交入口；若后台已处理，前台没有足够明确的“已处理”反馈。
  - 原因：`components/result/appeal-action-panel.tsx` 之前主要围绕 `submitted` 状态设计，没有完整覆盖 `reviewed` 的展示与锁定逻辑。
  - 解决：新增 `isReviewed`、`isLocked`、状态说明文案，并把 `appealedAt`、`appealUpdatedAt`、处理备注同步展示。
  - 当前结果：学生在结果页/历史详情页都能直接看到申诉是否已处理，且处理后不可重复提交。
- 后台题目详情页改动后，前台题库刷新不彻底。
  - 现象：后台能改题目，但从题目详情页执行启停或删除后，学生端 practice 链路可能读不到最新状态。
  - 原因：`app/admin/questions/[id]/page.tsx` 的 server action 只 revalidate 了后台路径，没有一起失效前台相关缓存。
  - 解决：统一使用 `revalidateQuestionPaths()`，把 admin 与 practice 相关路径一并刷新。
  - 当前结果：题目从后台详情页变更后，前台 `/practice`、`/practice/part1`、`/practice/part23` 能读取到同步后的状态。

## 4. 技术知识点
- **`revalidatePath`**：Next.js App Router 的缓存失效手段。今天的修正重点不是“能不能改成功”，而是“改完后哪些页面要一起刷新”，否则会出现后台已改、前台仍读旧数据的问题。
- **`router.refresh()`**：客户端提交申诉后，直接刷新当前路由对应的服务端数据，适合在不跳转页面的情况下拿到最新申诉状态。
- **Server Action**：`app/admin/questions/[id]/page.tsx` 中的启用/停用/删除逻辑放在服务端执行，适合配合 `redirect` 和 `revalidatePath` 做后台管理操作。
- **状态机思维**：申诉流程天然是 `none → submitted → reviewed`，前端 UI 需要围绕状态迁移设计文案、禁用条件和结果展示，而不是只处理“是否提交过”这一种情况。

## 5. 专业词汇
- **cache invalidation**：缓存失效。后台数据更新后，要主动让依赖该数据的页面重新取数。
- **`reviewed`**：申诉已处理状态，表示后台治理链路已经完成这条申诉的处理。
- **`isLocked`**：前端交互锁定条件；今天用于申诉已提交或已处理后禁用文本框和提交按钮。
- **`revalidateQuestionPaths`**：今天新增的统一路径刷新函数，用来保证题库后台和前台的缓存同步失效。
- **working tree**：当前本地工作区。今天没有新 commit，改动仍主要停留在 working tree 中。
